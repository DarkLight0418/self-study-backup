# 로직 트리 공부 관련 적용_20260418

오히려 공부처럼 **범위가 넓고, 막막하고, 어디서부터 손대야 할지 헷갈리는 작업**에 잘 맞습니다. 로직 트리(또는 이슈 트리)는 큰 문제를 더 작은 문제로 분해해서 다루는 방식이고, 좋은 트리는 보통 **겹치지 않으면서도 전체를 빠짐없이 덮는 구조(MECE)**를 지향합니다. 이런 구조화 방식은 실무 문제 해결뿐 아니라 복잡한 과제를 다룰 때도 유효하다고 설명됩니다. ([Management Consulted][1])

## 1) 먼저 본질부터 보면

공부에서 어려운 순간은 보통 이런 때입니다.

- “뭘 모르는지 모르겠다”
- “복습해야 하는데 양이 너무 많다”
- “문제를 틀렸는데 원인을 설명 못 하겠다”
- “열심히 했는데 점수가 왜 안 오르는지 모르겠다”

이때 그냥 “더 열심히”로 가면 감정만 커지고, 구조는 안 보입니다.
로직 트리 사고법은 이걸 **감정의 문제 → 구조의 문제**로 바꿔 줍니다. 즉, “못한다”가 아니라 “어느 가지(branch)에서 막혔는가?”로 바뀌는 것입니다. 이런 식의 구조화는 복잡한 문제를 다룰 때 특히 효과적입니다. ([Management Consulted][1])

## 2) 공부에 적용하면 무엇이 달라지나

핵심은 공부를 3종류의 문제로 나눠 보는 것입니다.

### A. 이해 문제

“개념을 정확히 이해하지 못함”

예:

- 용어 뜻을 모름
- 원리 연결이 안 됨
- 비슷한 개념끼리 구분이 안 됨

### B. 수행 문제

“알긴 아는데 문제에서 못 씀”

예:

- 문제 조건 해석 실패
- 풀이 절차가 떠오르지 않음
- 시간 압박에서 무너짐

### C. 관리 문제

“내용보다 공부 운영이 무너짐”

예:

- 복습 누락
- 오답 정리 안 됨
- 범위 분배 실패
- 집중력 소모 관리 실패

이렇게 나누면 “나는 머리가 나쁘다” 같은 뭉뚱그린 자기평가 대신, **이해·수행·관리 중 어디가 병목인지** 볼 수 있습니다. 로직 트리의 장점이 바로 이런 식의 분해와 우선순위화에 있습니다. ([Management Consulted][1])

## 3) 실제 적용 방법

공부에 적용할 때는 아래 순서가 가장 실용적입니다.

### 1단계. 맨 위 질문을 하나로 적기

질문은 모호하면 안 됩니다.

좋지 않은 질문:

- “왜 공부가 안 되지?”

좋은 질문:

- “정보처리기사 실기 점수가 안 오르는 이유는 무엇인가?”
- “네트워크 chapter 4를 문제에 적용하지 못하는 이유는 무엇인가?”
- “코딩 테스트에서 30분 이상 막히는 원인은 무엇인가?”

### 2단계. 1차 가지를 3~4개로 나누기

처음부터 너무 많이 나누면 오히려 망합니다.

예:
“네트워크 문제를 못 푼다”
→ 개념 이해 부족 / 문제 해석 부족 / 풀이 절차 미숙 / 암기 유지 실패

여기서 중요한 건 **같은 레벨끼리 나누는 것**입니다.
예를 들어 “TCP”, “집중력 부족”, “문제를 덜 풀음”을 한 줄에 섞으면 기준이 다릅니다. 과목 개념, 심리 상태, 학습량은 서로 다른 축이라 트리가 흐려집니다. MECE 원칙은 이런 혼선을 줄이기 위한 기준입니다. ([RocketBlocks][2])

### 3단계. 각 가지를 다시 “원인 단위”까지 내리기

예:
“문제 해석 부족”
→ 용어를 정확히 못 읽음 / 조건의 우선순위를 못 잡음 / 그림을 글로 못 바꿈

“암기 유지 실패”
→ 복습 간격이 김 / 인출 없이 재독만 함 / 유사개념 비교가 없음

여기까지 내려가야 해결책이 나옵니다.
트리는 **진단 도구**이지, 예쁘게 그리는 도구가 아닙니다.

### 4단계. 각 말단 가지에 대응 행동을 붙이기

예:

- 용어를 정확히 못 읽음 → 용어 카드 1장 1정의 작성
- 조건 우선순위를 못 잡음 → 문제문에 기호 표시
- 재독만 함 → 3분 인출 후 정답 비교
- 유사개념 비교 없음 → “TCP vs UDP” 같은 비교표 작성

이 순간 로직 트리는 단순 사고법이 아니라 **실행 설계도**가 됩니다.

## 4) 공부에 특히 잘 맞는 3가지 사용법

### 사용법 1. 개념 정리용

예를 들어 “운영체제 스케줄링”을 공부한다면

스케줄링
→ 선점형 / 비선점형
→ 선점형: RR, SRT, Priority
→ 비선점형: FCFS, SJF
→ 각 알고리즘: 기준 / 장점 / 단점 / 자주 나오는 함정

이 방식은 개념을 “줄글”이 아니라 “구조”로 기억하게 해 줍니다.

### 사용법 2. 오답 분석용

예:
“왜 이 문제를 틀렸는가?”
→ 개념 모름 / 계산 실수 / 조건 누락 / 문제 요구 오해 / 시간 부족

그리고 다시
“개념 모름”
→ 정의를 모름 / 예외 조건을 모름 / 비슷한 개념 혼동

이렇게 하면 오답장이 단순 기록이 아니라 **재발 방지 시스템**이 됩니다.

### 사용법 3. 시험 대비용

예:
“중간고사 10일 전 점수를 가장 많이 올릴 방법은?”
→ 과목별 점수 기여도 / 현재 취약도 / 회복 가능성 / 투자 대비 효율

그러면

- 비중 크고
- 지금 약하고
- 짧은 시간에 회복 가능한 파트

부터 먼저 치는 식으로 우선순위가 보입니다.
이건 무작정 분량 순서대로 보는 것보다 훨씬 전략적입니다. 문제 해결에서 구조화 후 우선순위를 정하는 접근과 맞닿아 있습니다. ([McKinsey & Company][3])

## 5) 예시 하나로 보여드리면

질문:
**“정보처리기사 실기에서 알고리즘 문제를 자꾸 틀리는 이유는?”**

로직 트리 예시:

알고리즘 문제를 틀린다
→ 개념 부족
→ 문제 해석 부족
→ 구현 절차화 부족
→ 검산 습관 부족

개념 부족
→ 최대공약수/재귀/배열 처리 원리 미흡
→ 시간복잡도 감 부족

문제 해석 부족
→ 입력·출력 형식 놓침
→ 요구사항을 중간에 바꿔 읽음

구현 절차화 부족
→ 손으로 단계 분해 안 함
→ 변수 역할을 먼저 정하지 않음

검산 습관 부족
→ 예제 입력 대입 안 함
→ 경계값 확인 안 함

이제 해결책은 자연스럽게 붙습니다.

- 개념 부족 → 개념 1개당 대표 문제 2개 연결
- 문제 해석 부족 → 문제문에 입력/처리/출력 표시
- 구현 절차화 부족 → 코드 전 30초 의사코드
- 검산 부족 → 예제 1회 + 극단값 1회

이렇게 되면 “열심히 하자”가 아니라 **어디를 고쳐야 하는지**가 보입니다.

## 6) 다만, 공부에 적용할 때의 한계도 있습니다

여기서 놓치기 쉬운 점이 있습니다.

첫째, 로직 트리는 **구조화 도구**이지 학습 자체를 대신하지는 않습니다.
트리를 잘 그려도, 인출 연습·문제 적용·반복 복습이 없으면 점수는 잘 안 오릅니다. 구조화는 방향을 잡아주고, 실력은 반복 훈련이 만듭니다. ([McKinsey & Company][3])

둘째, 너무 잘게 쪼개면 실행력이 떨어집니다.
트리가 4단계, 5단계로 지나치게 깊어지면 “분석만 하는 사람”이 되기 쉽습니다.

셋째, 감정 문제와 지식 문제를 구분해야 합니다.
예를 들어 집중 저하는 “개념 부족”이 아니라 “에너지 관리”의 가지일 수 있습니다. 이 둘을 섞으면 해결책이 어긋납니다.

## 7) 공부용으로 바꾸면 가장 좋은 형태

업무식 로직 트리를 공부용으로 바꾸면 저는 이 틀을 추천드립니다.

### 공부 로직 트리 기본형

**목표**
→ 무엇을 잘해야 하는가

**평가 기준**
→ 어떤 상태가 되면 잘한 것인가

**방해 요인**
→ 왜 아직 안 되는가

**개선 행동**
→ 그래서 무엇을 바꿀 것인가

예:
“네트워크 chapter 4를 시험에서 맞히기”
→ 목표: RDT, GBN, SR 핵심 차이 설명 가능
→ 평가 기준: 서술형 설명 + 그림 보고 상태 해석 가능
→ 방해 요인: 시퀀스 번호 축 이해 부족 / sender-receiver 역할 혼동 / ACK 흐름 미정리
→ 개선 행동: 상태도 직접 그리기 / GBN-SR 비교표 / 빈칸 인출 연습

이 틀은 단순 암기보다 훨씬 강합니다.

## 8) 정리

결론은 이렇습니다.

**로직 트리 사고법은 공부에 매우 잘 맞습니다.**
특히 아래 상황에서 강력합니다.

- 공부 범위가 넓어서 막막할 때
- 왜 틀리는지 원인을 모르겠을 때
- 시험 전 우선순위를 정해야 할 때
- 오답을 재발 방지 구조로 바꾸고 싶을 때

하지만 주의점도 있습니다.

- 트리 작성이 공부를 대신하진 않는다
- 너무 세분화하면 실행이 죽는다
- 구조화 뒤에는 반드시 인출·적용·오답 반복이 붙어야 한다

한 문장으로 압축하면:

**로직 트리는 공부 내용을 “정리”하는 도구가 아니라, 공부 실패 원인을 “분해”해서 개선 포인트를 찾는 도구로 쓸 때 가장 강합니다.**
