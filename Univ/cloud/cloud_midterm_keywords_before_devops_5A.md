## 0. 먼저 잡을 큰 흐름

- **클라우드 컴퓨팅** = 인터넷으로 컴퓨팅 자원 제공
- **클라우드 네이티브** = 클라우드의 장점을 극대화하도록 앱을 설계/개발/운영
- **핵심 축** = 컨테이너 + 마이크로서비스 + 오케스트레이션 + API + 자동화
- **운영 관점 핵심** = 확장성, 복원력, 이식성, 자동화, 관찰성
- **중요 사고 전환**
  - 서버를 직접 만지는 방식 -> 선언형/자동화 방식
  - 큰 하나의 앱 -> 작고 독립적인 서비스
  - 상태를 내부에 둠 -> 상태를 외부로 분리
  - 수동 배포/수동 운영 -> 자동 배포/자동 복구

## 1. 클라우드 컴퓨팅 vs 클라우드 네이티브

### 클라우드 컴퓨팅
- 인터넷 기반 리소스 제공 서비스
- 자원: 서버, 스토리지, 네트워크, DB
- **사용량 기반 비용 지불**
- 특징
  - 확장성
  - 온디맨드 프로비저닝
  - 비용 효율성
- 서비스 모델
  - **IaaS**: 인프라 제공
  - **PaaS**: 개발/배포 플랫폼 제공
  - **SaaS**: 완성 소프트웨어 제공
- 유형
  - 퍼블릭 / 프라이빗 / 하이브리드

### 클라우드 네이티브
- 클라우드에 **최적화된 앱 설계/개발/배포/운영 방식**
- 클라우드 장점 활용
  - 동적 확장성
  - 분산 처리
  - 복원력
- 구성 키워드
  - 컨테이너화
  - 마이크로서비스
  - 동적 오케스트레이션
  - API-First
  - CI/CD
  - 불변 인프라
- 특징
  - 민첩성
  - 확장성
  - 회복성
  - 이식성
  - 자동화

### 비교 한 줄
- **클라우드 컴퓨팅** = 클라우드 자원을 쓰는 것
- **클라우드 네이티브** = 클라우드에 맞게 앱 자체를 설계하는 것

## 2. 분산 시스템 / 12요소 앱 / SLA

### 분산 시스템
- 여러 컴퓨터가 네트워크로 연결된 전체 시스템
- 기대 효과
  - 안전성
  - 신뢰성
  - 고가용성
  - 경제성
- 흔한 착각
  - 네트워크는 항상 안정적이다 X
  - 지연은 무시 가능하다 X
  - 대역폭은 무한하다 X
  - 네트워크는 안전하다 X
  - 토폴로지는 고정이다 X
  - 관리자는 항상 하나다 X
  - 전송 비용은 공짜다 X

### CAP 이론
- **C** 일관성
- **A** 가용성
- **P** 네트워크 파티션 허용
- 파티션은 현실적으로 발생 가능
- SQL 쪽은 일관성 강조, NoSQL 쪽은 가용성 강조 흐름으로 자주 설명

### 12요소 앱 핵심 키워드
- 코드베이스 1개, 다양한 배포
- 종속성 명시적 선언
- 설정은 코드 밖 환경으로 분리
- backend service는 연결된 자원처럼 취급
- **build / release / run 분리**
- **stateless process**
- 서비스별 데이터 격리
- 동시성: 수평 확장
- 폐기 가능성: 빠른 시작/종료
- dev-prod 일치
- 로그 = 이벤트 스트림
- 관리 작업 = 일회성 프로세스

### SLA / SLO
- **SLA** = 서비스 수준 협약, 가용성 수치
- 여러 서비스 조합 시 **전체 가용성은 더 낮아짐**
- 예: 99.95% x 99.90% -> 99.85%
- **SRE** = 신뢰성/가용성 향상 관점

## 3. 쿠버네티스 핵심

### 왜 필요한가
- 도커만으로 실행은 가능
- 하지만 서비스가 커지면 운영 문제가 발생
  - 죽은 컨테이너 자동 복구 필요
  - 트래픽 증가 시 자동 확장 필요
  - 여러 서버에 배치 관리 필요
  - 무중단 배포 필요

### 정의
- **컨테이너 오케스트레이션 플랫폼**
- 자동 배포 / 확장 / 관리

### 핵심 구조
- **클러스터** = 여러 서버 묶음
- **컨트롤 플레인** = 관리/제어
- **워커 노드** = 실제 실행

### 컨트롤 플레인 구성 요소
- **API Server**: 프론트엔드
- **etcd**: 클러스터 데이터 저장소
- **Scheduler**: 파드 배치 노드 결정
- **Controller Manager**: 상태 감시, 원하는 상태 유지

### 워커 노드 구성 요소
- **kubelet**: 파드 실행/상태 관리
- **kube-proxy**: 네트워크 규칙/포워딩
- **container runtime**: 컨테이너 실행

### 핵심 오브젝트
- **Pod**: 최소 실행 단위
- **Service**: 고정 접근 주소, 로드밸런싱
- **ReplicaSet**: 파드 복제본 수 유지
- **Deployment**: 배포/업데이트/롤백
- **StatefulSet**: 순서/고유성/지속 스토리지 필요할 때
- **DaemonSet**: 모든 노드에 1개씩 실행
- **Job / CronJob**: 일회성 / 스케줄 작업
- **Volume**: 데이터 영속성
- **ConfigMap / Secret**: 설정 / 민감정보 분리
- **Ingress**: 외부 HTTP/HTTPS 라우팅

### 동작 원리
- 개발자가 **desired state 선언**
- 시스템이 현재 상태를 계속 감시
- 다르면 자동으로 맞춤
- 핵심 표현: **선언형 시스템**, **자동 복구**, **자동 확장**

### 장단점
- 장점: self-healing, auto-scaling, rolling update, 인프라 추상화
- 단점: 러닝 커브 큼, 설정 복잡, 소규모 서비스엔 과할 수 있음

### 한 줄 비유
- **도커 = 포장**
- **PM2 = 실행**
- **쿠버네티스 = 운영**

## 4. 컨테이너 / 서버리스 / 함수 / 마이크로서비스

### 컨테이너
- 앱 + 라이브러리 + 설정 + 실행환경을 묶은 배포 단위
- OS 커널 공유, 앱 간 격리
- 장점
  - 빠른 시작
  - 이식성
  - 높은 응집도
  - scale-out 적합
- 관련 키워드
  - 이미지
  - COW 파일 시스템
  - OCI
  - containerd
  - CRI

### 서버리스
- 인프라 관리는 클라우드 제공자가 담당
- 개발자는 코드 실행에 집중
- 사용한 만큼 비용 지불
- 형태
  - **FaaS**
  - **CaaS**

### 함수(FaaS)
- 짧은 수명
- 단순, 독립, 무상태 작업에 적합
- 이벤트 기반 처리
- 제약
  - 실행 시간 제한
  - 기능 하나에 집중
  - 의존성/환경 제약
  - 벤더 락인 위험
  - 온프레미스/자체 클러스터 실행 부적합한 경우 있음

### 마이크로서비스
- 작고 독립적
- 느슨한 결합, 높은 응집도
- 비즈니스 기능 단위 분리
- 팀 단위 소유/배포/운영
- 장점
  - 민첩성
  - 장애 격리
  - 독립 배포
  - 확장성
  - 관찰성 향상
- 과제
  - 분산 시스템 복잡성
  - 데이터 일관성
  - 네트워크 비용
  - 버전 호환성
  - 모니터링/로깅
  - 의존성 관리

### 함수 vs 서비스
- **함수**: 짧고 단순한 작업
- **서비스**: 오래 실행, 상태/하드웨어/유연성 필요

## 5. 클라우드 네이티브 앱 설계

### 설계 시 고려 요소
- 운영 효율성
- 보안
- 안정성
- 확장성
- 비용

### 운영 효율성
- 모든 것 자동화
- 모든 것 모니터링
- 모든 것 문서화
- 점진적 변화
- 장애 대비 테스트

### 보안(심층 방어)
- 소스 코드 저장소 보안
- 컨테이너 이미지 최소화
- 컨테이너 저장소 접근 제어
- 인증된 이미지 사용
- 클러스터 접근 보호
- RBAC 활성화
- 서비스 간 통신 보호
- 데이터 보호

### 안정성
- **reliability**: 장애가 있어도 동작 신뢰
- **availability**: 일정 시간 동안 사용 가능
- 수평 확장과 다중 인스턴스 중요

### 전통 구조 vs 클라우드 네이티브
- 전통 구조: 상태를 앱 인스턴스 내부에 두기 쉬움
- 클라우드 네이티브: **상태를 외부로 분리**
- 장애를 없애려 하기보다 **장애를 가정하고 설계**

### 상호작용 방식
- **오케스트레이션**: 중앙에서 흐름 제어
- **코레오그래피**: 이벤트/메시지 기반 느슨한 협력

### API 설계와 버저닝
- 전략
  - 단일 버전 강제
  - 포인트 투 포인트 다버전 유지
  - **하위 호환 유지가 가장 효율적**
- 버전 방식
  - 글로벌 버저닝
  - 리소스 버저닝
  - MIME 기반 버저닝
- 원칙
  - 기존 필드 이름 함부로 변경/삭제 금지
  - 기본값/선택값으로 하위 호환 고려
  - 롤백 시 상위 호환도 고려
- 시맨틱 버저닝
  - major: 호환성 깨짐
  - minor: 하위 호환 기능 추가
  - patch: 하위 호환 버그 수정

## 6. 서비스 커뮤니케이션 / 게이트웨이 / 이그레스 / 서비스 메시

### 트래픽 방향
- **East-West**: 클러스터 내부 서비스 간
- **North-South**: 외부 클라이언트 <-> 내부 서비스

### 통신 방식
- 동기식 요청/응답
- 비동기식 큐/이벤트
- pub/sub

### 동기식 문제
- 쓰레드 대기
- 전체 지연 증가
- 연쇄 장애

### 비동기식 장점
- 느슨한 결합
- 최종적 일관성 확보 쉬움
- 장애 전파 감소

### 프로토콜 키워드
- HTTP
- WebSocket
- HTTP/2
- gRPC
- MQTT
- AMQP
- JSON vs Protocol Buffers

### 멱등성
- 같은 요청 여러 번 처리해도 결과 동일
- 재시도/중복 메시지 환경에서 중요
- 고유 ID로 중복 제거

### 게이트웨이
- 클라이언트 요청의 진입점
- 기능
  - SSL 오프로딩
  - 인증/권한
  - 라우팅
  - 정적 자원 전달
  - rate limit
  - retry
  - circuit breaker
  - 캐시/압축/로깅
- 구현 예
  - Nginx, HAProxy, Envoy

### Ingress / Egress
- **Ingress**: 외부 -> 내부 서비스 라우팅
- **Egress**: 내부 -> 외부 서비스 접근 제어

### 서비스 메시
- 서비스 간 공통 통신 기능 분리
- 프록시 계층으로 트래픽/보안/장애처리/관측성 담당
- 기능
  - 트래픽 관리
  - 재시도
  - 타임아웃
  - circuit breaker
  - mTLS
  - tracing
  - metrics
- 대표 도구 키워드
  - Istio
  - Linkerd
  - Consul Connect

## 7. 데이터 다루기 핵심

### 스토리지 유형
- **오브젝트 스토리지**: 대용량 파일, 저렴, 확장성 큼
- **파일 스토리지**: 공유 파일 시스템
- **블록 스토리지**: DB/OS 디스크, 빠른 I/O
- **아카이브 스토리지**: 장기 보관, 매우 저렴, 느림
- **인메모리 스토리지**: 초고속, 휘발성, 고비용

### 데이터베이스 유형
- 키/값
- 문서형
- 관계형
- 그래프
- 컬럼 패밀리
- 시계열
- 검색 인덱스

### 선택 기준
- 기능 요구
  - 데이터 형식
  - 읽기/쓰기 패턴
  - 관계 구조
  - 일관성 모델
  - 동시성
  - 검색/인덱싱
- 비기능 요구
  - 성능
  - 확장성
  - 가용성
  - 백업/복구
  - 비용
  - 팀 경험

### CDC / ETL / 데이터 레이크
- **CDC**: 변경 사항 추적
  - 알림
  - 캐시 무효화
  - 감사 로그
  - 검색 인덱스 갱신
  - 분석
- **ETL**: 추출 -> 변환 -> 적재
- **데이터 레이크**: 분석/리포팅용 통합 저장
- 주의
  - 서비스 DB를 분석팀이 직접 읽으면 결합도 증가

### 클라이언트 데이터 접근
- 제한된 토큰(발렛 키)
- GraphQL

### 확장 키워드
- **샤딩**: 수평 분할
- **캐싱**: TTL, CDC 기반 무효화, 백그라운드 갱신
- **CDN**: 사용자 가까운 위치에서 콘텐츠 제공, 지연 감소, 가용성 향상

## 8. 시험 직전 비교 포인트

### 1) 클라우드 컴퓨팅 vs 클라우드 네이티브
- 클라우드 컴퓨팅 = 자원 사용 모델
- 클라우드 네이티브 = 앱 설계/운영 모델

### 2) 도커 vs 쿠버네티스
- 도커 = 컨테이너 생성/실행
- 쿠버네티스 = 다수 컨테이너 운영/배포/복구/확장

### 3) Pod vs Container
- 컨테이너 = 실행 단위
- 파드 = 컨테이너를 감싸는 쿠버네티스 최소 배포 단위

### 4) Deployment vs ReplicaSet
- ReplicaSet = 개수 유지
- Deployment = 배포 전략/업데이트/롤백 관리

### 5) StatefulSet vs DaemonSet
- StatefulSet = 순서/고유성/지속성 필요
- DaemonSet = 모든 노드에 하나씩

### 6) 함수 vs 서비스
- 함수 = 짧고 단순, 이벤트 중심
- 서비스 = 장시간 실행, 더 유연

### 7) 동기 vs 비동기
- 동기 = 단순하지만 지연/연쇄장애 취약
- 비동기 = 느슨한 결합, 확장성과 복원력 유리

### 8) JSON vs Protocol Buffers
- JSON = 가독성 좋음, 무겁다
- protobuf = 빠르고 작음, 스키마/생성기 필요

### 9) Queue vs Pub/Sub
- Queue = 한 소비자가 처리
- Pub/Sub = 여러 구독자가 처리

### 10) Reliability vs Availability
- reliability = 장애에도 제대로 동작
- availability = 사용 가능한 시간 비율

## 9. 마지막 암기 체크리스트

- IaaS / PaaS / SaaS 구분 가능?
- 클라우드 네이티브 구성요소 5개 이상 바로 말할 수 있는가?
- CAP의 C/A/P 뜻 설명 가능한가?
- 12요소 앱 핵심 문장 3~4개로 요약 가능한가?
- 쿠버네티스의 control plane / worker node 역할 구분 가능한가?
- Pod / Service / Deployment / StatefulSet / DaemonSet 차이 설명 가능한가?
- 함수가 적합한 경우와 부적합한 경우 구분 가능한가?
- 마이크로서비스 장점과 과제 둘 다 말할 수 있는가?
- API 버저닝과 하위 호환성 중요성 설명 가능한가?
- 동기식 통신의 문제 3가지 말할 수 있는가?
- 멱등성이 왜 필요한지 설명 가능한가?
- 서비스 메시가 왜 필요한지 말할 수 있는가?
- 스토리지 5종류의 용도 구분 가능한가?
- CDC / ETL / 샤딩 / 캐싱 / CDN 키워드 연결 가능한가?
