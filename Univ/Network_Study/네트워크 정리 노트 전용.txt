2. 보안(security) - 종단간 암호화

p14
1. TCP
- 신뢰 가능한 정보 전달, 연결형
- 혼잡 제어
- 제공 X : 타이밍, 최소 처리량 보장, 보안(위 레이어에서 씌움)
- 클라이언트 - 서버 간 연결 지향 설정 필요

2. UDP
- 정보 전달 신뢰성 보장 안됨.
- 제공 X : flow control, 혼잡 제어, throughput guarantee, 보안, 연결 설정
- 왜 있을까? : TCP는 모든 정보 전달로 인해 망사용료를 많이 잡아먹음, 한계량 초과 및 속도 저하
  (단 넷플릭스는 화질을 포기하지 않으려고 TCP를 씀)

p19 [HTTP(HyperText Transport Protocol) 개요]
웹의 응용(Application) Layer 프로토콜
- 클라이언트/서버 모델
  - 클라이언트 : 요청/수신 후 표시하는 브라우저
  - 서버 : HTTP를 통해 클라이언트 객체 요청이 들어올 때 서버를 통해 전송

p20 [HTTP를 간단히 살펴보자면]
- HTTP는 TCP를 가짐. 또한 상태가 없음(Stateless)
(즉, 뒤로 가기를 누르면 URL 뒷 페이지를 다시 가져옴)

- 포트넘버 80, HTTP 클라이언트는 브라우저, 
HTTP 서버는 웹 서버를 통해 응용 계층 메시지 교환 후 종료

!!알아두기!!
프로토콜은 상태 유지 자체가 매우 어려움. 서로 충돌 시 유지 X, 조정되어야 함.

p21 [비연결형 HTTP VS 연결형 HTTP]
- 비연결형(Non-persistent)
  - 최소한 하나의 객체를 보내야 하는 것이 특징
  - 다시 보내게 되면 다시 오픈 후 전송 및 종료

- 연결형(Persistent)
  - 서버를 통해 TCP 연결을 열었을 때, 단일 클라이언트 & 서버 간 단일 TCP 연결을 통해 전송

p22~23 [비연결형 HTTP 예시(|||)]

HTTP 요청 메시지는 TCP 소켓에 담아서 보내고,
서버는 받은 후 응답 객체를 TCP 소켓에 담아서 보낸다.


- HTML 파일 받기 위해 1번 연결
- 이미지 10개 받기 위해 10번 연결
- 총 매우 많은 연결 생성/종료 발생

파일 하나 받을 때마다 TCP 연결을 새로 맺고, 응답받고, 끊는다.

p24(시간 없을 경우 넘겨도 됨)

- RTT: 작은 패킷이 클라이언트에서 서버까지 갔다가 다시 돌아오는 시간
- 비연결형 HTTP response time = 2RTT + 파일 전송 시간
- 객체 당 2RTT 필요

p25 [연결형 HTTP(HTTP 1.1)]

- 단 하나의 RTT만큼 참조

E-mail, SMTP, IMAP


p49~50
[E-mail]
3개의 메이저 컴포넌트
- user agents -> 메일 이용 도구
- mail servers -> 서버
- SMTP(Simple Mail Transfer Protocol)

1. User Agent -> 메일 리더

2. Mail Servers
-> 메일함에는 사용자에게 들어오는 메시지가 포함되어 있음
-> 메시지 큐는 순차적으로 메일 메시지를 내보냄
-> SMTP 프로토콜은 메일 메시지 전송과 메일 서버를 통해 받는 것을 관리

p51 [E-mail: the RFC]
- E-mail은 무결성 훼손 방지를 위해 TCP를 씀
- direct Transfer : 전송 서버(클라이언트처럼 행동)를 수신 서버로

- 3단계 Transfer
1. handshaking (greeting) -> TCP에서 필요
2. transfer of messages
3. 클로저(Closure)

- command/response 상호작용
  - commands : ASCII text
  - response : 상태 코드 및 구절
- 메시지는 무조건 7-bit ASCI 형식


p52 [예제 앨리스 -> 밥]
```
나(보내는 사람) -> 나의 서버 -> 받는 사람 서버 -> 받는 사람
```

즉 - 각각 서버를 거치는 구조

p56 [Mail message format]

SMTP : 이메일 메시지 교환 - RFC 531 (HTTP 같은)
이메일 메시지를 위한 문맥 정의 - RFC 822 (HTML 같은)

header에는 To, From, Subject(제목) 포함
body에는 ASCII 문자로 이루어진 메시지 저장

p57 [Mail Access Protocol]

DNS(Domain Name System)
p59
- 많은 이름 서버의 계층 구조로 구현된 분산 데이터베이스
- 응용 계층 프로토콜 중 하나
- IP <-> 이름 간 매칭 필요

- 로드 밸런싱 관리 : 1대1 / "다대1"

p60 [DNS Services, Structure]
DNS Service : 호스트 이름을 IP 주소로 변환

로드 밸런싱을 통한 부하 분산

** 왜 DNS를 하나로 하면 안되는가? **
- "Single Point Of Failure" : SPOF
- 어마무시하게 많은 트래픽
- 거리가 있는 중앙 데이터베이스
- 유지보수
- 스케일의 부족(매일 6천억번 DNS Query가 오고감)

-> 그러므로 DNS는 UDP 사용 (잘못 요청하면 다시 보내면 됨)
