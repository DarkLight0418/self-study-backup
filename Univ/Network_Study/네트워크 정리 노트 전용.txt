p12
전송/공유 서비스 요소 4가지(앱에 있어 필요한)
1. 데이터 무결성(data integrity)
2. 보안(security) - 종단간 암호화
3. 시간(timing) - 싱크, 실시간 소통
4. 대역폭, 통과 가능한 양 -> 효율성 (throughput)

p13
일반적인 앱들의 전송 서비스 요구 정도
- 파일 다운/업, 이메일, 웹 문서 : TCP(유실되면 안됨), 대역폭 유동적, 시간 고려 x
- 오디오, 비디오(real-time, streaming), interactvie games: UDP(조금 유실되도 큰 문제 없음)
- 텍스트 메시지 교환: TCP, 시간 고려 o

p14
1. TCP
- 신뢰 가능한 정보 전달, 연결형
- 혼잡 제어
- 제공 X : 타이밍, 최소 처리량 보장, 보안(위 레이어에서 씌움)
- 클라이언트 - 서버 간 연결 지향 설정 필요

2. UDP
- 정보 전달 신뢰성 보장 안됨.
- 제공 X : flow control, 혼잡 제어, throughput guarantee, 보안, 연결 설정
- 왜 있을까? : TCP는 모든 정보 전달로 인해 망사용료를 많이 잡아먹음, 한계량 초과 및 속도 저하
  (단 넷플릭스는 화질을 포기하지 않으려고 TCP를 씀)

p19
HTTP(HyperText Transport Protocol) 개요
웹의 응용(Application) Layer 프로토콜
- 클라이언트/서버 모델
  - 클라이언트 : 요청/수신 후 표시하는 브라우저
  - 서버 : HTTP를 통해 클라이언트 객체 요청이 들어올 때 서버를 통해 전송

p20
HTTP를 간단히 살펴보자면
- HTTP는 TCP를 가짐. 또한 상태가 없음(Stateless)
(즉, 뒤로 가기를 누르면 URL 뒷 페이지를 다시 가져옴)

- 포트넘버 80, HTTP 클라이언트는 브라우저, 
HTTP 서버는 웹 서버를 통해 응용 계층 메시지 교환 후 종료

프로토콜은 상태 유지 자체가 매우 어려움. 서로 충돌 시 유지 X, 조정되어야 함.

p21

비연결형 HTTP VS 연결형 HTTP
- 비연결형(Non-persistent)
  - 최소한 하나의 객체를 보내야 하는 것이 특징
  - 다시 보내게 되면 다시 오픈 후 전송 및 종료

- 연결형(Persistent)
  - 서버를 통해 TCP 연결을 열었을 때, 단일 클라이언트 & 서버 간 단일 TCP 연결을 통해 전송

p22~23 [비연결형 HTTP 예시(|||)]

HTTP 요청 메시지는 TCP 소켓에 담아서 보내고,
서버는 받은 후 응답 객체를 TCP 소켓에 담아서 보낸다.


- HTML 파일 받기 위해 1번 연결
- 이미지 10개 받기 위해 10번 연결
- 총 매우 많은 연결 생성/종료 발생

파일 하나 받을 때마다 TCP 연결을 새로 맺고, 응답받고, 끊는다.

p24(시간 없을 경우 넘겨도 됨)

- RTT: 작은 패킷이 클라이언트에서 서버까지 갔다가 다시 돌아오는 시간
- 비연결형 HTTP response time = 2RTT + 파일 전송 시간
- 객체 당 2RTT 필요

p25 [연결형 HTTP(HTTP 1.1)]

- 모든 걸 한번 열고 쫙 보내는 형태
- 단 하나의 RTT만큼 참조
