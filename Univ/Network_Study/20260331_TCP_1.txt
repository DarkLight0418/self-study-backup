DevRTT - RTT의 변동 정보

DevRTT = (1-베타)*DevRTT + 베타*|Sample RTT - EstimatedRTT |

TCP는 네트워크 환경에 따라 타임아웃을 자동 적용함(변동될 수 있음)

---

TCP Sender
TCP는 특정 상황마다 정해진 동작 수행
1. 미확인 패킷을 기준으로 하나 존재
2. 타이머는 하나만 존재
3. 타임아웃 시 해당 데이터 세그먼트 재전송, 타이머 재시작
4. ACK 수신
  - ACK 받으면 내부 상태 업데이트
  - 확인 X 데이터 -> 타이머 유지

ACK generation(TCP receiver action)
receiver ACK 생성
-> 500ms로 묶어서 보냄, 딜레이 ACK
-> 2개 정도의 ACK가 연속된 패킷이 도착하면 즉시 누적된 ACK를 보냄

확인된다 -> 피기백된다 잘 전송됐다

순서에 어긋난 패킷 -> sender에게 duplicate ACK 보냄
바로 패킷을 제대로 받을 경우 -> 바로 ACK를 생성해서 보냄, 상태를 업데이트하게 함

---
시나리오(p.6)
1번째 그림 -> ACK 손실 : 리시버 동일한 데이터 2번 받게됨
2번째 그림 -> 타임아웃 발생 전에 잘 전송된 케이스, 중복상황까지 설계한 그림

타임아웃이 발생했더라도 이미 나는 120번까지 다시 받은 걸로 보내주면, 중복까지 고려하여 처리

---
장점
특정 번호까지 다 확인, (문제 발생했어도) 전체 통신에서는 큰 문제가 발생하지 않음

TCP fast retransmit

TCP 타임아웃이 발생하기까지 기다림 -> 느릴 수 있음.
동일한 ACK가 동작하면 -> 문제가 발생했구나! -> 재전송
