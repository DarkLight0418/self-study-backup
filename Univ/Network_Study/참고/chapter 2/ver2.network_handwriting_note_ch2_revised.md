# 0. 먼저 외울 핵심 문장 15개

1. **응용 계층 프로토콜은 메시지 타입, 문법, 의미, 송수신 규칙을 정의한다.**
2. **프로세스 식별은 IP만으로 부족하고, IP + Port가 필요하다.**
3. **소켓은 응용 계층 메시지를 전송 계층으로 내보내는 문이다.**
4. **앱이 보는 전송 서비스 핵심은 무결성, 보안, 시간, 처리량이다.**
5. **TCP는 신뢰적·연결형이고, UDP는 단순·비연결형이다.**
6. **기본 TCP/UDP 소켓은 암호화가 없어서 TLS가 필요하다.**
7. **HTTP는 TCP를 쓰지만 기본적으로 stateless다.**
8. **쿠키는 stateless한 HTTP 위에서 상태를 유지하는 장치다.**
9. **비연결형 HTTP는 객체마다 TCP 연결을 새로 만든다.**
10. **연결형 HTTP는 하나의 TCP 연결로 여러 객체를 보낸다.**
11. **웹 캐시는 가까운 곳에서 응답해 지연과 트래픽을 줄인다.**
12. **SMTP는 메일을 보내는 프로토콜이고, IMAP은 메일을 가져오는 프로토콜이다.**
13. **DNS는 이름을 IP로 바꾸는 분산 계층형 데이터베이스다.**
14. **P2P의 핵심 장점은 참여자가 늘면 자원도 같이 늘어난다는 점이다.**
15. **DASH는 현재 대역폭에 맞춰 화질을 고르는 적응형 스트리밍이다.**

# 1. Application Layer 큰 그림 (p8 ~ p16)

## p8 프로세스끼리 메시지로 통신
- 네트워크 앱 = 서로 다른 호스트에서 실행되는 프로그램이 **메시지 교환** 하는 것
- 같은 호스트 내부 통신은 OS가 처리
- 다른 호스트 간 통신은 네트워크로 메시지를 주고받음

### 손필기 감각
- **프로그램은 말을 하고, 네트워크는 그 말을 다른 호스트로 보낸다.**

---

## p9 소켓의 개념
- 응용 계층 메시지를 **소켓에 담아 보냄**
- 소켓 = 응용과 전송 계층 사이의 출입문
- 보내는 쪽 소켓 1개, 받는 쪽 소켓 1개가 대응

### 한 줄 암기
- **소켓은 문, 메시지는 문을 통과하는 짐**

||| 소켓 그림
- 프로세스 -> 소켓 -> 전송/네트워크 -> 소켓 -> 프로세스

---

## p10 주소와 프로세스 식별
- 집 주소(IP)만 알면 부족
- 같은 집(호스트) 안에 여러 사람(프로세스)이 있을 수 있기 때문
- 그래서 **IP 주소 + 포트 번호** 둘 다 필요

### 시험식 표현
- **메시지를 받은 뒤 누구에게 줄지 분류하려면 포트 번호가 필요하다.**
- 예: `10.10.16.68:80`
- 포트 번호는 다른 IP에서 같을 수 있음

### 손필기 감각
- **집 주소만 있으면 집까지만 감. 누구 방인지 알려면 포트가 필요함.**

---

## p11 응용 계층 프로토콜이 정의하는 것
응용 계층 프로토콜은 아래를 정한다.

1. **메시지 타입**
   - request, response
2. **메시지 문법**
   - 필드가 어떻게 생겼는가
3. **메시지 의미**
   - 각 필드의 뜻
4. **송수신 규칙**
   - 언제 보내고, 어떻게 응답하는가

### 프로토콜 종류
- **Open protocol**
  - RFC에 공개
  - 예: HTTP, SMTP
- **Proprietary protocol**
  - 특정 기업/서비스 전용

---

## p12 앱이 필요로 하는 전송 서비스 4요소
1. **data integrity**
   - 데이터가 안 깨져야 함
2. **security**
   - 종단 간 암호화, 무결성, 인증
3. **timing**
   - 지연이 낮아야 함
4. **throughput**
   - 충분한 전송량 필요

### 시험식 감각
- 파일 전송: 무결성 중요
- 음성/게임: 시간 중요
- 영상: 처리량 중요
- 로그인/인증: 보안 중요

---

## p13 일반 앱의 요구사항 감각
### TCP 쪽 감각
- 파일 다운로드/업로드
- 이메일
- 웹 문서
- 텍스트 메시지

공통점:
- **유실되면 안 됨**
- 대역폭은 유동적이어도 됨
- 실시간성은 덜 중요

### UDP 쪽 감각
- 실시간 오디오/비디오
- 인터랙티브 게임

공통점:
- **조금의 손실은 감수 가능**
- 대신 **지연이 더 중요**

---

## p14 TCP vs UDP
## 1) TCP
- 신뢰적 전송
- 연결형
- flow control
- congestion control
- 제공하지 않는 것:
  - 타이밍 보장
  - 최소 처리량 보장
  - 자체 보안

## 2) UDP
- 비연결형
- 신뢰성 보장 X
- flow control X
- congestion control X
- throughput guarantee X
- 보안 X
- 연결 설정 X

### 왜 UDP가 있나?
- TCP는 절차가 많고 무거움
- 항상 정확성 위주로 가면 지연과 비용이 커질 수 있음
- 손실을 조금 허용하고 빠르게 보내야 하는 앱도 있음

### 손필기식 표현
- **TCP = 정확하게**
- **UDP = 빨리, 단순하게**

---

## p16 TLS
### Vanilla TCP/UDP 소켓의 문제
- 암호화 없음
- 비밀번호도 평문으로 오갈 수 있음

### 그래서 필요한 것: TLS
- TCP 연결 암호화
- 데이터 무결성
- 종단 인증

### 연결 메모
- **HTTPS = HTTP + TLS**
- TCP 자체 보안이 아니라, **위에서 TLS를 씌우는 것**

---

# 2. Web & HTTP (p18 ~ p47)

## p18 Web 페이지와 객체
- 웹 페이지는 여러 **객체(object)** 로 구성
- 객체는 HTML, 이미지, 오디오 등
- 하나의 페이지는 base HTML + 여러 참조 객체로 이루어질 수 있음
- 각 객체는 URL로 지정 가능

### 손필기 감각
- **웹 페이지 1개를 여는 건, 실제로 여러 파일을 가져오는 것**

---

## p19 HTTP 개요
- HTTP = Web의 응용 계층 프로토콜
- 클라이언트/서버 모델
  - 클라이언트: 브라우저
  - 서버: 웹 서버
- 브라우저가 요청하고, 서버가 객체를 응답

---

## p20 HTTP의 성격
- HTTP는 **TCP 사용**
- 포트 번호: **80**
- 브라우저와 웹 서버가 HTTP 메시지 교환
- 그리고 중요한 특징: **stateless**

### stateless
- 이전 요청을 기억하지 않음
- 뒤로 가기를 눌러도, 예전 화면을 다시 요청해서 가져오는 구조

### 손필기 포인트
- **상태 유지가 어렵다 = 프로토콜 설계는 단순하지만, 사용자 상태를 따로 저장해야 함**

---

## p21 비연결형 vs 연결형 HTTP
## Non-persistent HTTP
- TCP 연결 열기
- 객체 1개 보내기
- TCP 연결 닫기

## Persistent HTTP
- TCP 연결 1번 열기
- 여러 객체를 같은 연결로 전송
- RTT 절약
- 연결 생성/종료 오버헤드 감소

### 핵심 비교
- **비연결형 = 객체마다 새 연결**
- **연결형 = 하나의 연결로 여러 객체**

---

## p22 ~ p23 비연결형 HTTP 예시
- HTML 파일 받기 위해 연결 1번
- 이미지 10개 받기 위해 연결 10번
- 총 매우 많은 연결 생성/종료 발생

### 손필기식 표현
- **파일 하나 받을 때마다 TCP 연결을 새로 맺고, 응답받고, 끊는다.**

||| 비연결형 HTTP 흐름 그림
- URL 입력
- TCP 연결
- HTML 요청/응답
- 연결 종료
- 이미지마다 반복

---

## p24 응답 시간
- RTT = 작은 패킷이 클라이언트에서 서버까지 갔다가 돌아오는 시간
- 비연결형 HTTP에서 객체 1개 응답 시간:
  - **2 RTT + 파일 전송 시간**

### 암기 포인트
- 1 RTT: TCP 연결 성립
- 1 RTT: HTTP 요청 + 첫 응답 바이트
- 그 뒤: 파일 전송 시간

---

## p25 연결형 HTTP(HTTP/1.1)
- 한 번 열고 여러 객체를 연달아 전송
- 비연결형보다 효율적
- 연결 오버헤드 감소
- 체감적으로는 **쭉 이어서 받는 구조**

### 손필기 표현
- **한번 열고 쫙 보내는 형태**

---

## p26 ~ p29 HTTP 메시지
## 요청 메시지
- request line
- header lines
- 필요 시 body

대표 메서드:
- **GET**: 이거 줘
- **POST**: 이거 담아서 보낼게
- **HEAD**: 헤더만 보여줘
- **PUT**: 이거 갱신/업로드할게

## 응답 메시지
- status line
- header lines
- data(body)

### 손필기 메모
- Date 헤더는 **서로 시간 기준(synch)을 맞추는 데도 의미가 있음**

||| HTTP general format 그림
- request line / header / blank line / body
- response line / header / blank line / data

---

## p30 HTTP 상태 코드
- **200 OK**
- **301 Moved Permanently**
- **400 Bad Request**
- **404 Not Found**
- **505 HTTP Version Not Supported**

### 손필기 표현
- 301 = 영구 이동, 리다이렉트
- 404 = 서버에 문서 없음

## p31 ~ p35 쿠키
### 왜 필요한가?
- HTTP는 stateless
- 그런데 로그인, 장바구니, 세션 유지 같은 것은 상태 정보가 필요

### 그래서 쿠키 사용
쿠키 관련 4요소:
1. 응답 메시지의 set-cookie
2. 다음 요청 메시지의 cookie
3. 브라우저의 cookie 파일
4. 서버의 백엔드 DB

### 저장 위치
- 클라이언트 PC의 웹 브라우저

### 쿠키 용도
- 자동 로그인 비슷한 권한 유지
- 장바구니
- 유저 세션 정보
- 추천

### 중요한 구분
- **쿠키로 재로그인하는 게 아니라**
- **쿠키를 통해 이미 인증된 상태를 식별/권한 부여하는 것**

### 프라이버시 메모
- 편리하지만 추적 문제 존재
- 광고/추적 쿠키는 보안·사생활 측면에서 주의

||| 쿠키 동작 그림
- 첫 방문 -> 서버가 ID 생성 -> 브라우저 저장
- 다음 요청 -> cookie ID 포함 -> 서버 DB와 대조

---

## p36 ~ p42 웹 캐시와 Conditional GET
## p36 웹 캐시(프록시 서버)
- 자주 안 바뀌는 객체를 미리 저장
- 캐시에 있으면 캐시가 바로 반환
- 없으면 캐시가 origin 서버에서 받아와 전달

### 손필기식 흐름
if 캐시에 객체 존재  
-> 캐시가 고객에게 객체 반환  
else  
-> 캐시가 origin 서버에서 받아와 저장 후 반환

### 왜 하는가?
- 응답 시간 단축
- 액세스 링크 트래픽 감소
- 원본 서버 부담 감소

### ISP 메모
- 보통 이런 캐시는 ISP/학교/회사 쪽에 설치됨

---

## p42 Conditional GET
목표:
- 캐시에 최신 버전이 있으면 **다시 안 보내기**

방법:
- 요청 헤더: `If-Modified-Since`
- 응답:
  - 변경 없음 -> `304 Not Modified`
  - 변경됨 -> `200 OK` + 새 데이터

### 한 줄 요약
- **안 바뀐 파일은 안 보내서 시간과 트래픽을 아낀다**

---

## p43 ~ p47 HTTP/2, HTTP/3
## HTTP/1.1 문제
- 요청 순서대로 처리하는 FCFS 성격
- 앞의 큰 객체 때문에 뒤의 작은 객체가 기다릴 수 있음
- 이것이 **HOL blocking**

## HTTP/2
- 객체를 프레임 단위로 나눔
- 프레임 전송 순서를 더 유연하게 조절
- 큰 파일 뒤에 작은 파일이 너무 오래 묶이지 않게 완화

### 손필기 감각
- **왜 하지? -> 유저 편의성**
- **프레임별로 잘라 보냄**

## HTTP/3
- 더 보안성이 강화된 방향
- UDP 기반에서 오류/혼잡 제어를 포함하는 진화형
- 이후 QUIC과 연결해서 이해

||| HOL blocking 완화 그림
- HTTP/1.1: O1 큰 객체 뒤에 O2/O3/O4 대기
- HTTP/2: 프레임 섞어서 O2/O3/O4 먼저 체감 전달

# 3. E-mail / SMTP / IMAP (p49 ~ p57)

## p49 ~ p50 이메일의 3대 구성
1. **User Agent**
   - 메일 리더
   - 메일 작성/읽기 도구
2. **Mail Server**
   - 수신 메일함 보관
   - 송신 대기열 보관
3. **SMTP**
   - 메일 서버 간 메일 전송

### 손필기 감각
- **사람은 UA를 쓰고, 실제 메일 전달은 서버끼리 처리한다**

---

## p51 SMTP
- E-mail은 무결성이 중요해서 **TCP 사용**
- 포트 번호: **25**
- 전송 서버가 클라이언트처럼 행동해서 받는 서버로 보냄

### 3단계
1. handshaking
2. message transfer
3. closure

### 특징
- command / response 상호작용
- commands = ASCII text
- response = 상태 코드 + 구절
- 7-bit ASCII 형식

---

## p52 Alice -> Bob 시나리오
흐름:
`보내는 사람 -> 내 메일 서버 -> 상대 메일 서버 -> 받는 사람`

### 한 줄 요약
- **메일은 사람끼리 바로 가는 게 아니라, 각자 메일 서버를 거친다**

---

## p55 SMTP vs HTTP
### 공통점
- ASCII command/response
- 상태 코드 존재

### 차이점
- **HTTP = pull**
- **SMTP = push**

### 손필기 표현
- HTTP는 가져오는 느낌
- SMTP는 밀어 넣는 느낌

---

## p56 메일 메시지 형식
- SMTP는 메일을 교환하는 규칙
- 메일 내용 자체의 형식은 별도로 정의됨

구성:
- header
  - To
  - From
  - Subject
- blank line
- body

### 주의
- 메일 본문의 To/From 과
- SMTP의 MAIL FROM, RCPT TO 는 다름

---

## p57 Mail access protocol
- SMTP: 메일을 **보내는 것**
- IMAP: 서버에 저장된 메일을 **가져오는 것**
- HTTP: 웹메일 인터페이스(Gmail 등)

### 반드시 구분
- **SMTP = 송신**
- **IMAP = 수신/조회/관리**

---

# 4. DNS (p59 ~ p72)

## p59 DNS란?
- 사람은 이름을 기억함
- 네트워크는 IP 주소로 동작함
- 그래서 이름 <-> IP 변환이 필요

### 정의
- **많은 이름 서버의 계층 구조로 구현된 분산 데이터베이스**
- 동시에 **응용 계층 프로토콜**

---

## p60 DNS를 왜 하나로 모으면 안 되나?
중앙집중형 DNS의 문제:
- **SPOF**
- 엄청난 트래픽
- 멀리 있는 중앙 DB
- 유지보수 어려움
- 확장성 부족

### 손필기 메모
- 너무 많은 질의가 발생하므로 중앙집중형은 스케일이 안 됨
- DNS는 그래서 **분산형 계층 구조** 로 감

### 서비스
- hostname -> IP
- host aliasing
- mail server aliasing
- load distribution

---

## p61 ~ p64 DNS 계층 구조
1. **Root DNS**
2. **TLD DNS**
   - .com, .org, .edu, .kr ...
3. **Authoritative DNS**
   - 조직이 실제로 보유한 DNS
4. **Local DNS**
   - 학교/회사/ISP의 기본 DNS

### 손필기 표현
- Local은 계층에 엄격히 속하지는 않지만, 사용자는 보통 여기부터 시작

||| DNS 계층 구조 그림
Local -> Root -> TLD -> Authoritative

---

## p65 반복적 질의(Iterative Query)
- 내가 모르면 "저쪽에 물어봐" 하고 다음 서버를 알려줌

### 손필기 표현
- **내가 DNS 모름 -> 쟤한테 물어봐**

---

## p66 재귀적 질의(Recursive Query)
- 요청받은 쪽이 대신 끝까지 찾아줌

### 손필기 감각
- 사용자는 편함
- 대신 중간 DNS 서버 부담이 커질 수 있음

---

## p67 캐싱과 TTL
- DNS 결과는 캐시됨
- TTL이 지나면 사라짐
- 캐시 덕분에 루트 서버 부담 감소
- 단, IP가 바뀌면 TTL이 끝나기 전까지는 구 정보가 남을 수 있음

### 암기 포인트
- **TTL = 살아있는 시간**
- TTL이 끝나면 다시 DNS 질의

---

## p68 DNS 레코드
- **A**: hostname -> IP
- **NS**: authoritative name server
- **CNAME**: 별칭 -> 진짜 이름
- **MX**: 메일 서버

### 꼭 구분
- A는 주소
- MX는 메일
- CNAME은 별칭
- NS는 권한 서버

---

## p69 ~ p70 DNS 메시지
- query와 reply는 비슷한 형식
- header + question + answer + authority + additional 정보

### 시험 팁
- 여기 세부 비트 필드보다
- **질문/응답/권한/추가정보 구조**를 기억하는 편이 효율적

---

## p71 DNS 등록
- 도메인 등록 시 registrar에 authoritative DNS 정보 제공
- registrar가 TLD 쪽에 NS, A 레코드 삽입
- 이후 자신의 authoritative DNS에서 A, MX 등을 관리

---

## p72 DNS 보안
문제:
- DDoS
- redirect attack
- DNS poisoning

보완:
- **DNSSEC**

### 손필기 메모
- DNS는 핵심 인프라라 공격 대상이 되기 쉬움

---

# 5. P2P / BitTorrent (p74 ~ p82)

## p74 P2P 구조
- 항상 켜져 있는 중앙 서버 없음
- peer들이 직접 통신
- 서비스도 받고, 제공도 함

### 장점
- 참여자가 늘면 자원도 같이 늘어남
- **self scalability**

### 단점
- IP 변경
- 접속/이탈(churn)
- 관리 복잡

---

## p75 ~ p78 파일 배포 시간 관점
## Client-Server
- 서버 혼자 N명에게 파일 복사
- N이 커질수록 서버 부담 증가

## P2P
- 각 peer도 업로드에 참여
- 사용자 증가가 곧 자원 증가로도 이어짐

### 시험 감각
- **P2P의 핵심 장점 = 참가자가 많아질수록 공급 능력도 올라감**

---

## p79 ~ p82 BitTorrent
- 파일을 작은 chunk로 나눔
- tracker가 참여 peer 목록 관리
- peer는 서로 chunk 교환

### 요청 전략
- **rarest first**
  - 가장 희귀한 조각부터 요청

### 전송 전략
- **tit-for-tat**
  - 나에게 잘 보내는 peer에게 우선 전송
- 주기적으로 optimistic unchoke 수행

### 손필기 감각
- **비토렌트는 조각 분할 + 희귀 조각 우선 + 상호이익 교환 구조**

---

# 6. Streaming / DASH / CDN (p84 ~ p98)

## p84 왜 어려운가?
- 비디오 트래픽이 매우 큼
- 사용자 수가 많음
- 사용자 환경이 제각각
  - 유선/무선
  - 빠름/느림

### 결론
- 단일 서버로는 감당 어려움
- 분산된 응용 계층 인프라 필요

---

## p85 ~ p86 비디오와 압축
- 비디오는 연속된 이미지
- 공간적 중복, 시간적 중복을 이용해 압축

### 개념
- **spatial coding**
  - 한 장면 내부 중복 제거
- **temporal coding**
  - 이전 프레임과의 차이만 전송

### 비트레이트
- CBR: 고정
- VBR: 가변

---

## p87 ~ p90 스트리밍의 어려움
문제:
- 대역폭 변화
- 지연/jitter
- 손실
- 재생은 끊기면 안 됨

해결:
- **플레이아웃 버퍼링**
- 클라이언트가 미리 받아 저장 후 재생

### 손필기 한 줄
- **조금 미리 받아놔야, 네트워크 흔들려도 영상이 안 끊김**

---

## p91 ~ p92 DASH
- 서버:
  - 비디오를 chunk로 나눔
  - 여러 비트레이트 버전 저장
  - manifest file 제공
- 클라이언트:
  - 현재 대역폭 측정
  - 가능한 최대 화질 chunk 선택
  - 상황 따라 화질을 바꿔가며 요청

### 핵심
- **좋은 망 = 고화질**
- **나쁜 망 = 저화질**
- 즉, 적응형 스트리밍

---

## p93 ~ p98 CDN
### 왜 필요한가?
단일 mega-server 문제:
- 병목
- 먼 거리
- 단일 장애점
- 확장성 부족

### CDN의 아이디어
- 여러 지역에 콘텐츠 복사본 저장
- 사용자 가까운 서버가 전송

### 손필기 표현
- **콘텐츠를 사용자 가까이에 미리 놔두는 구조**

### CDN 접근 흐름
- 사용자가 콘텐츠 요청
- DNS/CNAME 등을 통해 적절한 CDN 서버로 유도
- 가까운 복사본에서 전송

### Netflix 감각
- 등록/계정 쪽 서버와
- 실제 비디오를 뿌리는 CDN 서버 역할이 구분됨

---

# 7. Socket Programming (p100 ~ p109)

## p100 소켓 프로그래밍 목표
- 클라이언트/서버 앱이 실제로 소켓으로 통신하는 구조 이해

---

## p101 두 가지 소켓
- **UDP 소켓**
  - unreliable datagram
- **TCP 소켓**
  - reliable byte stream

---

## p102 ~ p105 UDP 소켓
### 특징
- 연결 설정 없음
- handshake 없음
- 보낼 때마다 목적지 IP와 포트 명시
- 순서 보장 X
- 손실 가능

### 응용 계층 관점
- UDP는 **datagram 단위** 전달

### 기억 포인트
- sender가 주소 붙여서 보냄
- receiver는 받은 패킷에서 상대 주소를 확인

||| UDP client/server 흐름 그림
- client socket 생성
- sendto()
- server recvfrom()
- server sendto()
- client recvfrom()

---

## p106 ~ p109 TCP 소켓
### 특징
- 먼저 연결 설정 필요
- 서버는 welcoming socket 준비
- 클라이언트가 접속하면 서버는 **그 클라이언트 전용 새 소켓** 생성
- 신뢰적, 순서 보장된 byte stream

### 기억 포인트
서버:
- bind
- listen
- accept

클라이언트:
- connect

이후:
- send / recv / close

### 한 줄 비교
- **UDP = 편지 묶음**
- **TCP = 연결된 파이프**

||| TCP client/server 흐름 그림
- server socket 생성 -> listen -> accept
- client socket 생성 -> connect
- request / response
- close

---

# 8. 헷갈리는 비교만 다시 정리

## 8-1 TCP vs UDP
| 항목 | TCP | UDP |
|---|---|---|
| 연결 설정 | O | X |
| 신뢰성 | O | X |
| 순서 보장 | O | X |
| 흐름 제어 | O | X |
| 혼잡 제어 | O | X |
| 사용 감각 | 정확성 우선 | 지연/단순성 우선 |

---

## 8-2 Non-persistent vs Persistent HTTP
| 항목 | Non-persistent | Persistent |
|---|---|---|
| TCP 연결 | 객체마다 새로 | 하나로 여러 객체 |
| RTT 부담 | 큼 | 줄어듦 |
| 효율 | 낮음 | 높음 |

---

## 8-3 HTTP vs SMTP
| 항목 | HTTP | SMTP |
|---|---|---|
| 기본 동작 | pull | push |
| 역할 | 웹 객체 요청/응답 | 메일 전송 |
| 대표 특징 | stateless | 서버 간 메일 전달 |

---

## 8-4 SMTP vs IMAP
| 항목 | SMTP | IMAP |
|---|---|---|
| 역할 | 보냄 | 가져옴/관리 |
| 사용 위치 | 메일 전송 | 메일 접근 |
| 감각 | 송신 | 수신/조회 |

---

## 8-5 DNS query 방식
| 항목 | Iterative | Recursive |
|---|---|---|
| 느낌 | 저 서버에 물어봐 | 내가 대신 찾아줄게 |
| 부담 | 요청자 쪽 | 중간 서버 쪽 |

# 9. 손필기용 최종 압축본

## 응용 계층 기본
- 프로세스끼리 메시지 교환
- 소켓 = 응용과 전송 계층 사이 문
- 프로세스 식별 = IP + Port
- 프로토콜 = 타입/문법/의미/규칙

## 전송 서비스
- integrity / security / timing / throughput
- TCP = 신뢰적, 연결형
- UDP = 단순, 빠름, 비연결형
- TLS = TCP 위 암호화

## HTTP
- TCP 사용, 포트 80
- stateless
- Non-persistent = 객체마다 연결
- Persistent = 한 연결로 여러 객체
- 쿠키 = 상태 유지
- 웹 캐시 = 가까운 곳에서 응답
- Conditional GET = 안 바뀌면 다시 안 보냄
- HTTP/2 = 프레임으로 HOL 완화
- HTTP/3 = UDP 기반 진화형

## E-mail
- UA / Mail Server / SMTP
- SMTP = 보냄, TCP 25, push
- IMAP = 가져옴

## DNS
- 이름 <-> IP 변환
- 분산 계층형 DB
- Root / TLD / Authoritative / Local
- A / NS / CNAME / MX
- 캐싱, TTL

## P2P
- 중앙 서버 없이 peer끼리 교환
- 참여자 증가 = 자원 증가
- BitTorrent = chunk / rarest first / tit-for-tat

## Streaming
- 대역폭 변화, 지터, 손실 문제
- buffer로 완충
- DASH = 상황 따라 화질 선택
- CDN = 가까운 서버에서 전송

## Socket
- UDP = sendto / recvfrom
- TCP = connect / accept
- UDP는 datagram
- TCP는 byte stream
