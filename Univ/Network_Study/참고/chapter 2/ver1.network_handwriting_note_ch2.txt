0. 시험 전에 먼저 외울 12문장

1. **응용 계층 프로토콜은 메시지 타입, 문법, 의미, 송수신 규칙을 정의한다.**
2. **프로세스 식별은 IP 주소만으로는 부족하고, 포트 번호까지 필요하다.**
3. **소켓은 응용 프로세스와 전송 계층 사이의 문이다.**이전 채팅 기록에 보시면 네트워크 얘기를 같이 한 효율적인 노트 정리법 자료가 있을
4. **TCP는 신뢰적이고 연결지향적이며, flow control / congestion control을 제공한다.**
5. **UDP는 비연결형이며 빠르고 단순하지만 신뢰성을 보장하지 않는다.**
6. **HTTP는 TCP를 사용하지만 기본적으로 stateless다.**
7. **비지속 HTTP는 객체마다 연결이 필요하고, 지속 HTTP는 하나의 연결로 여러 객체를 보낸다.**
8. **쿠키는 stateless한 HTTP 위에서 상태를 유지하기 위한 장치다.**
9. **SMTP는 메일을 보내는 프로토콜이고, IMAP은 서버에 저장된 메일을 가져오는 프로토콜이다.**
10. **DNS는 이름을 IP로 바꾸는 분산 계층형 데이터베이스다.**
11. **Flow control은 수신자 보호, congestion control은 네트워크 보호다.**
12. **TCP 혼잡 제어의 핵심 감각은 "조심스럽게 늘리고, 문제 생기면 크게 줄인다(AIMD)"이다.**
이전 채팅 기록에 보시면 네트워크 얘기를 같이 한 효율적인 노트 정리법 자료가 있을

1. 응용 계층 큰 그림

#1-1. 네트워크 앱이란?
- 서로 다른 **종단 시스템(end system)** 에서 실행되는 프로그램들이
- 네트워크를 통해 **메시지** 를 주고받는 것
- 핵심 장치:
  - **프로세스**
  - **소켓**
  - **응용 계층 프로토콜**
  - **전송 계층 서비스(TCP/UDP)**

#1-2. 프로세스와 소켓
- 프로세스 = 호스트 안에서 실행 중인 프로그램
- 서로 다른 호스트의 프로세스는 **메시지 교환** 으로 통신
- 소켓 = 응용이 메시지를 밀어 넣고 꺼내는 **출입문**

##손필기 한 줄
- **프로세스가 말하고, 소켓이 내보내고, 전송 계층이 옮긴다.**

#1-3. 프로세스 식별
- IP 주소만으로는 부족
- 같은 호스트에도 여러 프로세스가 동시에 존재 가능
- 따라서 식별자 = **IP 주소 + Port 번호**

##예시
- HTTP 서버: `:80`
- 메일 서버: `:25`

[그림]  
`클라이언트 프로세스 -> 소켓 -> 네트워크 -> 소켓 -> 서버 프로세스`

---

2. 응용 계층 프로토콜이 정의하는 것

응용 계층 프로토콜은 다음 4가지를 정의한다.

1. **메시지 타입**
   - request / response
2. **메시지 문법(syntax)**
   - 필드가 어떻게 생겼는가
3. **메시지 의미(semantics)**
   - 각 필드가 무엇을 뜻하는가
4. **송수신 규칙**
   - 언제 보내고, 어떻게 응답하는가

#프로토콜 종류
- **Open protocol**
  - RFC에 공개
  - 예: HTTP, SMTP
- **Proprietary protocol**
  - 특정 서비스/기업 전용

---

3. 앱이 전송 계층에 요구하는 것

앱이 중요하게 보는 4가지:

1. **data integrity**
   - 데이터가 안 깨져야 함
2. **timing**
   - 지연이 낮아야 함
3. **throughput**
   - 일정한 전송량이 필요할 수 있음
4. **security**
   - 암호화, 무결성, 인증

#감각적으로 구분하기
- **파일 다운로드 / 웹 문서 / 이메일**
  - 손실 허용 X
  - 보통 TCP
- **실시간 음성 / 게임**
  - 약간의 손실은 허용
  - 지연이 더 중요
  - UDP가 어울릴 수 있음

---

4. TCP vs UDP vs TLS

#4-1. TCP
- **신뢰적 전송**
- **연결지향**
- **flow control**
- **congestion control**
- 제공하지 않는 것:
  - 타이밍 보장
  - 최소 처리량 보장
  - 자체 보안

#4-2. UDP
- **비연결형**
- 신뢰성 보장 X
- 순서 보장 X
- 흐름 제어 X
- 혼잡 제어 X
- 대신 **단순하고 빠름**

#4-3. TLS
- 기본 TCP/UDP 소켓은 암호화 없음
- 비밀번호도 평문으로 갈 수 있음
- 그래서 **TLS** 로 TCP 연결 위에 보안 추가
  - 암호화
  - 데이터 무결성
  - 종단 인증

##손필기 비교표
구분 / TCP / UDP
연결 설정 / O / X
신뢰성 / O / X
순서 보장 / O / X
flow control / O / X
congestion control / O / X
속도/단순성 / 상대적으로 무거움 / 가볍고 단순

5. HTTP

#5-1. HTTP의 본질
- Web의 응용 계층 프로토콜
- **클라이언트-서버 모델**
  - 클라이언트 = 브라우저
  - 서버 = 웹 서버
- **TCP 사용**
- 기본 포트: **80**
- 중요한 특징: **stateless**

##stateless란?
- 서버가 이전 요청을 기억하지 않음
- 모든 요청이 독립적임

##왜 중요?
- 단순하고 확장에 유리
- 하지만 로그인/장바구니 같은 **상태 유지** 가 어려움

---

#5-2. 비지속 HTTP vs 지속 HTTP

##비지속(Non-persistent)
- TCP 연결 1번 열기
- 객체 1개 전송
- 연결 종료
- 객체가 많으면 연결도 많이 필요

##지속(Persistent, HTTP/1.1)
- TCP 연결 1번 열기
- 여러 객체를 같은 연결로 전송
- RTT 절약
- 오버헤드 감소

##핵심 비교
- **비지속 HTTP**: 객체마다 연결
- **지속 HTTP**: 한 연결로 여러 객체

##응답 시간 감각
- 비지속 HTTP에서 객체 1개당 대략:
  - **2 RTT + 파일 전송 시간**

[그림]  
- 왼쪽: HTML 1개 + 이미지 10개 = 연결 11개  
- 오른쪽: 하나의 연결에서 여러 객체 전송

---

#5-3. HTTP 메시지 구조

##요청 메시지
- Request line
- Header lines
- (필요하면) body

대표 메서드:
- **GET**: 가져와
- **POST**: 데이터 담아 보냄
- **HEAD**: 헤더만 요청
- **PUT**: 지정 URL의 자원 업로드/교체

##응답 메시지
- Status line
- Header lines
- Data(body)

대표 상태 코드:
- **200 OK**
- **301 Moved Permanently**
- **400 Bad Request**
- **404 Not Found**
- **505 HTTP Version Not Supported**

---

#5-4. 쿠키(Cookies)
HTTP는 stateless이므로, 상태 유지를 위해 쿠키 사용

구성 요소 4개:
1. HTTP 응답의 cookie header
2. 다음 HTTP 요청의 cookie header
3. 브라우저의 cookie 파일
4. 서버의 백엔드 DB

##쿠키 용도
- 인증/인가
- 장바구니
- 추천
- 세션 유지

##주의
- 편리하지만 프라이버시 문제도 있음

##한 줄 요약
- **쿠키는 stateless HTTP 위에 상태를 얹는 장치다.**

---

#5-5. 웹 캐시(Web Cache, Proxy)
- 브라우저가 요청을 캐시로 보냄
- 캐시에 있으면 바로 반환
- 없으면 캐시가 원 서버(origin)에서 받아와 저장 후 반환

##왜 쓰는가?
- 응답 시간 감소
- 트래픽 감소
- 원 서버 부담 감소

##핵심 포인트
- 캐시는 **클라이언트 입장에선 서버**, 원 서버 입장에선 **클라이언트**

---

#5-6. Conditional GET
목표:
- 캐시에 최신 버전이 있으면 **객체를 다시 보내지 않기**

방법:
- 요청: `If-Modified-Since`
- 응답:
  - 안 바뀜 -> `304 Not Modified`
  - 바뀜 -> `200 OK` + 데이터

##한 줄 요약
- **바뀌지 않은 파일은 다시 안 보내서 지연과 트래픽을 줄인다.**

---

#5-7. HTTP/2와 HTTP/3
##HTTP/1.1 문제
- 요청 순서대로 처리(FCFS)
- 큰 객체가 앞에 있으면 뒤의 작은 객체가 기다림
- 이것이 **HOL blocking**

##HTTP/2
- 객체를 **프레임 단위** 로 쪼개서 전송
- 전송 순서를 더 유연하게 조절
- HOL blocking 완화

##HTTP/3
- UDP 기반 위에서 더 발전
- 보안과 지연 측면 개선 방향

##시험 포인트
- **HTTP/2 = 프레임화로 HOL 완화**
- **HTTP/3 = UDP 기반 발전형**

6. E-mail: SMTP / IMAP / HTTP

#6-1. 이메일의 큰 구조
3대 구성:
1. **User Agent**
   - 메일 읽기/쓰기 도구
2. **Mail Server**
   - 메일함, 송신 대기열 보관
3. **SMTP**
   - 서버 간 전송

#6-2. SMTP
- 메일을 **보내는** 프로토콜
- TCP 사용
- 포트 **25**
- 3단계:
  1. handshaking
  2. message transfer
  3. closure

##특징
- command/response 방식
- ASCII 기반
- **push 방식**
- persistent connection 사용

##흐름
`보내는 사람 UA -> 보내는 쪽 메일 서버 -> 받는 쪽 메일 서버 -> 받는 사람 UA`

#6-3. IMAP / HTTP
- **IMAP**
  - 서버에 저장된 메일을 가져옴
  - 삭제/폴더 관리 가능
- **HTTP**
  - 웹메일 인터페이스(Gmail 등)

##핵심 비교
- **SMTP = 보낸다**
- **IMAP = 가져온다**

7. DNS

#7-1. DNS란?
- 사람은 이름을 기억
- 네트워크는 IP 주소를 사용
- DNS는 이 둘을 연결

##정의
- **분산된 계층형 데이터베이스**
- 동시에 **응용 계층 프로토콜**

#7-2. DNS를 중앙집중형으로 못 하는 이유
- SPOF(single point of failure)
- 트래픽 폭증
- 멀리 있는 중앙 DB
- 유지보수 어려움
- 확장성 부족

#7-3. 계층 구조
1. **Root DNS**
2. **TLD DNS**
   - `.com`, `.org`, `.edu`, `.kr` 등
3. **Authoritative DNS**
   - 실제 도메인 정보를 가진 서버
4. **Local DNS**
   - ISP / 학교 / 회사의 기본 DNS

[그림]  
`Local -> Root -> TLD -> Authoritative`

#7-4. DNS가 제공하는 것
- hostname -> IP 변환
- host aliasing
- mail server aliasing
- load distribution

#7-5. DNS 질의 방식
##Iterative query
- "난 몰라. 저 서버에 물어봐."

##Recursive query
- 요청받은 서버가 대신 끝까지 찾아줌

#7-6. 캐싱과 TTL
- DNS 결과는 캐시됨
- TTL이 지나면 만료
- 캐시 덕분에 루트 서버 부담 감소
- 단, IP가 바뀌면 즉시 전 세계 반영은 안 될 수 있음

#7-7. DNS 레코드
- **A**: hostname -> IP
- **NS**: 해당 도메인의 authoritative name server
- **CNAME**: 별칭 -> 진짜 이름
- **MX**: 메일 서버

##손필기 한 줄
- **DNS는 단순 전화번호부가 아니라, 계층형 분산 전화번호 시스템이다.**

8. P2P, BitTorrent

#8-1. P2P 구조
- 항상 켜져 있는 중앙 서버가 없음
- peer끼리 직접 통신
- 요청도 하고 제공도 함

##장점
- 참여자가 늘면 자원도 같이 늘어남
- self-scalability

##단점
- 관리 복잡
- IP 변경
- 접속/이탈(churn)

#8-2. 파일 배포 시간 관점
##Client-Server
- 서버 혼자 N명에게 보내야 해서
- 사용자 수 N이 커지면 부담이 큼

##P2P
- 각 peer도 업로드에 참여
- 사용자 수가 늘어도 서비스 능력도 증가

##시험 감각
- **P2P의 핵심 장점 = 참여자가 많아질수록 공급 능력도 늘어남**

#8-3. BitTorrent
- 파일을 작은 chunk로 나눔
- tracker가 참여 peer 목록 관리
- peer는 서로 chunk를 주고받음

##요청 전략
- **rarest first**
  - 가장 희귀한 조각부터 받음

##전송 전략
- **tit-for-tat**
  - 나에게 잘 보내는 peer에게 우선적으로 보냄
- optimistically unchoke도 존재

##한 줄 요약
- **BitTorrent는 조각 분할 + 희귀 조각 우선 + 상호 이익 교환 구조다.**

9. 스트리밍, DASH, CDN

#9-1. 스트리밍의 어려움
- 대역폭이 계속 변함
- 지연(jitter) 발생
- 패킷 손실 가능
- 재생은 끊기면 안 됨

#9-2. 플레잉 버퍼
- 클라이언트가 미리 받아서 버퍼에 저장
- 네트워크 지연 변동을 흡수
- 끊김을 줄임

#9-3. DASH
- Dynamic Adaptive Streaming over HTTP
- 서버가 비디오를 **여러 chunk / 여러 비트레이트** 로 저장
- 클라이언트가 현재 대역폭 보고 적절한 화질 chunk 요청

##핵심
- **대역폭 좋으면 고화질**
- **대역폭 나쁘면 저화질**
- 즉, 적응형 스트리밍

#9-4. CDN
- 전 세계 여러 위치에 콘텐츠 복사본 저장
- 사용자와 가까운 서버가 전송

##왜 필요한가?
- 하나의 mega server는
  - 병목
  - 멀리 있는 사용자 지연
  - 확장성 한계
  - SPOF 위험

##한 줄 요약
- **CDN은 콘텐츠를 사용자 가까이에 미리 두는 구조다.**

10. Socket Programming 핵심

#10-1. 목표
- 클라이언트/서버 애플리케이션을 직접 만드는 것

#10-2. UDP 소켓
- 연결 설정 없음
- 각 패킷에 목적지 IP/포트 직접 붙임
- 순서 보장 X, 손실 가능

##기억 포인트
- `sendto()`, `recvfrom()`
- 주소 정보가 같이 다님

#10-3. TCP 소켓
- 먼저 연결 설정 필요
- 서버는 welcoming socket 생성
- 클라이언트 접속 시 서버는 **새 연결 소켓** 생성
- 신뢰적, 순서 보장된 byte stream

##기억 포인트
- 서버:
  - bind
  - listen
  - accept
- 클라이언트:
  - connect
- 이후:
  - send / recv
  - close

##한 줄 비교
- **UDP는 편지 묶음(datagram), TCP는 연결된 파이프(byte stream)**

11. TCP 심화

#11-1. RTT와 Timeout
##RTT
- 패킷이 갔다가 ACK가 돌아오는 왕복 시간
- 고정값이 아니라 네트워크 상태 따라 변함

##왜 EstimatedRTT가 필요한가?
- SampleRTT는 튈 수 있음
- 그래서 부드럽게 평균낸 추정치 사용

##식
- `EstimatedRTT = (1-a) * EstimatedRTT + a * SampleRTT`
- 보통 최근 값보다 과거 흐름도 같이 반영

##TimeoutInterval
- `TimeoutInterval = EstimatedRTT + 4 * DevRTT`

##DevRTT
- RTT가 얼마나 흔들리는지 나타냄

##감각
- RTT 평균만 보는 게 아니라,
- **변동 폭까지 보고 안전 마진을 둔다**

---

#11-2. ACK와 재전송
- TCP sender는 미확인 세그먼트 기준으로 동작
- 타이머는 하나를 중심으로 운용
- timeout 나면 해당 세그먼트 재전송

##ACK 관련 포인트
- delayed ACK 가능
- out-of-order 세그먼트가 오면 duplicate ACK 가능
- timeout까지 기다리면 느리므로 **fast retransmit** 사용

##fast retransmit 핵심 감각
- 같은 ACK가 반복되면
- "중간 세그먼트가 빠졌을 가능성이 크다"
- timeout 전에 재전송

---

#11-3. Flow Control
##목적
- **수신자 버퍼 보호**
- 수신자가 처리 가능한 만큼만 보내게 함

##핵심 변수
- **rwnd (receive window)**

##의미
- 수신 버퍼에 남아 있는 공간
- 송신자는 이 값을 보고 아직 ACK 안 된 데이터 총량을 조절

##한 줄 요약
- **Flow control = receiver가 sender를 조절하는 것**

---

#11-4. Connection Management
##왜 handshake가 필요한가?
- 그냥 바로 보내면 상태 확인이 안 됨
- 오래된 패킷, 중복 패킷, half-open connection 문제가 생김

##3-way handshake
1. SYN
2. SYN + ACK
3. ACK

##의미
- "연결할래?"  
- "좋아, 너도 나도 준비됨"  
- "확인, 시작하자"

##종료
- 종료는 양방향이 독립적
- FIN / ACK 기반으로 닫힘

---

#11-5. Congestion Control
##본질
- Flow control과 다름
- 수신자가 못 받는 문제가 아니라,
- **네트워크 자체가 감당 못하는 상태**

##혼잡이 생기면
- 라우터 큐 증가
- 지연 증가
- 손실 증가
- 재전송 증가
- 다시 더 혼잡해짐

##핵심 인식
- sender는 라우터 내부 상태를 직접 모름
- 결과(지연, 손실)를 보고 **추정** 해야 함

---

#11-6. AIMD
- **AI(Additive Increase)**: 천천히 증가
- **MD(Multiplicative Decrease)**: 손실 시 크게 감소

##감각
- 네트워크 여유가 있는 것 같으면 조금씩 올리고
- 문제 생기면 절반 수준으로 확 줄임

##왜 좋은가?
- 안정성
- 효율성
- 공정성

---

#11-7. cwnd와 Slow Start
##cwnd
- congestion window
- 송신자가 동시에 네트워크에 띄워둘 수 있는 양

##Slow Start
- 초기에는 `cwnd = 1 MSS`
- RTT마다 대략 2배로 증가
- 빠르게 가능한 용량 탐색

##이후
- 손실이 나거나 ssthresh에 도달하면
- 더 천천히 증가

##감각 정리
- **처음엔 빠르게 탐색**
- **문제 생기면 줄이고 조심스럽게 증가**

12. 가장 헷갈리는 비교 정리

#12-1. Flow Control vs Congestion Control
항목 / Flow Control / Congestion Control
보호 대상 / 수신자 / 네트워크
원인 / receiver 버퍼 부족 / 라우터/링크 혼잡
핵심 변수 / rwnd / cwnd
핵심 질문 / "상대가 더 받을 수 있나?" / "망이 더 버틸 수 있나?"

#12-2. SMTP vs IMAP
항목 / SMTP / IMAP
역할 / 메일 보냄 / 메일 가져옴/관리
방향 / 송신 중심 / 수신/접근 중심
핵심 감각 / push / 서버 저장 메일 접근

#12-3. HTTP vs SMTP
항목 / HTTP / SMTP
기본 방식 / pull / push
대표 용도 / 웹 객체 요청/응답 / 메일 전송
상태 / 기본 stateless / 세션 대화는 존재하나 웹처럼 쓰이지 않음
메시지 / 객체별 응답 / 메일 메시지 전송

#12-4. Non-persistent vs Persistent HTTP
항목 / Non-persistent / Persistent
연결 수 / 객체마다 별도 / 하나로 여러 객체
RTT 부담 / 큼 / 감소
효율 / 낮음 / 높음

#12-5. TCP vs UDP
항목 / TCP / UDP
연결 / O / X
신뢰성 / O / X
순서 보장 / O / X
혼잡/흐름 제어 / O / X
사용 감각 / 정확성 우선 / 지연/단순성 우선

13. 손필기 최종 압축본 (정말 짧게)

#응용 계층
- 프로세스끼리 메시지 교환
- 소켓 = 앱과 전송 계층 사이 문
- 프로세스 식별 = IP + Port
- 프로토콜 = 메시지 타입/문법/의미/규칙

#전송 계층 선택
- TCP = 신뢰성, 연결형, 흐름 제어, 혼잡 제어
- UDP = 빠르고 단순, 신뢰성 없음
- TLS = TCP 위 보안

#HTTP
- TCP 사용, stateless
- Non-persistent: 객체마다 연결
- Persistent: 하나의 연결로 여러 객체
- 쿠키 = 상태 유지
- 캐시 = 가까운 곳에서 응답
- Conditional GET = 안 바뀐 객체 재전송 방지
- HTTP/2 = 프레임으로 HOL 완화
- HTTP/3 = UDP 기반 발전형

#E-mail
- UA / Mail Server / SMTP
- SMTP = 보냄, TCP 25, push
- IMAP = 가져옴

#DNS
- 이름 <-> IP 변환
- 분산 계층형 DB
- Root / TLD / Authoritative / Local
- 캐싱 + TTL
- 레코드 A, NS, CNAME, MX

#P2P / 스트리밍
- P2P = 참여자 늘면 자원도 증가
- BitTorrent = chunk, rarest first, tit-for-tat
- DASH = 상황 따라 화질 조절
- CDN = 사용자 가까운 서버에서 제공

#TCP 심화
- RTT는 변함 -> EstimatedRTT 사용
- Timeout = EstimatedRTT + 4DevRTT
- fast retransmit = 중복 ACK 보고 빨리 재전송
- flow control = rwnd, 수신자 보호
- 3-way handshake = 양방향 상태 확인
- congestion control = 네트워크 보호
- AIMD = 조금씩 증가, 크게 감소
- cwnd / slow start 중요
